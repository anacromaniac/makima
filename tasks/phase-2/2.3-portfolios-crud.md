# Task 2.3: Portfolios CRUD

## Objective
Implement full CRUD for portfolios with user ownership isolation.

## Steps
1. Create migration `migrations/0003_portfolios.sql`:
   - `portfolios` table per schema in MVP spec.
   - Index on `user_id`.
2. Create `crates/db/src/repositories/portfolio_repo.rs`:
   - `create(new_portfolio) -> Result<Portfolio>`
   - `find_by_id(id) -> Result<Option<Portfolio>>`
   - `find_by_user_id(user_id, pagination) -> Result<PaginatedResult<Portfolio>>`
   - `update(id, name, description) -> Result<Portfolio>`
   - `delete(id) -> Result<()>` (cascade deletes transactions via FK)
3. Create `crates/api/src/portfolios/`:
   - `dto.rs` — CreatePortfolioRequest (name required, description optional, garde validated), UpdatePortfolioRequest (name, description), PortfolioResponse.
   - `handlers.rs`:
     - `GET /api/v1/portfolios` — list user's portfolios (paginated).
     - `POST /api/v1/portfolios` — create portfolio. Sets base_currency to EUR.
     - `GET /api/v1/portfolios/{id}` — get portfolio. Verify ownership (user_id matches AuthUser). Return 404 if not found or not owned.
     - `PUT /api/v1/portfolios/{id}` — update. Verify ownership.
     - `DELETE /api/v1/portfolios/{id}` — delete. Verify ownership.
4. Ownership check: every handler that takes a portfolio ID must verify it belongs to the authenticated user. If it doesn't, return 404 (not 403, to avoid leaking existence of other users' portfolios).
5. Annotate all endpoints with utoipa macros.

## Done criteria
- CRUD operations work correctly.
- User A cannot see, modify, or delete user B's portfolios (returns 404).
- Paginated list returns correct `total_items` and `total_pages`.
- Creating portfolio with empty name returns validation error.
- Deleting a portfolio cascade-deletes its transactions (tested when transactions exist in later tasks).
- Endpoint behavior is covered in `crates/api/tests/portfolios.rs` using the shared API integration harness.
- All clippy/fmt/doc/test checks pass.
