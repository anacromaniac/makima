# Task 2.6: Positions

## Objective
Implement the positions endpoint that calculates positions on-the-fly from transactions, with closed position support.

## Steps
1. No migration needed — positions are derived, not persisted.
2. Create `crates/db/src/repositories/position_repo.rs`:
   - `get_positions(portfolio_id) -> Result<Vec<RawPosition>>` — SQL query that aggregates transactions per asset: SUM(quantity) for buys, SUM(quantity) for sells, weighted average cost. Joins with assets table for asset details. Returns all assets with any transaction history in the portfolio.
3. Create position types in `crates/domain/src/models/position.rs` (if not already present):
   - `Position` — asset (Asset), quantity (Decimal), average_cost (Decimal), current_price (Option<Decimal>), current_value (Option<Decimal>), gain_loss_absolute (Option<Decimal>), gain_loss_percentage (Option<Decimal>), closed (bool)
4. Create `crates/api/src/positions/`:
   - `dto.rs` — `PositionResponse` with all calculated fields.
   - `handlers.rs`:
     - `GET /api/v1/portfolios/{id}/positions` — verify portfolio ownership. Fetch aggregated positions from DB. For each position, look up latest price from `price_history` (if available). Calculate gain/loss using domain logic from task 1.2. Mark positions with quantity = 0 as closed. Support `?show_closed=true/false` query parameter (default: false).
5. Use the gain/loss calculation functions from `crates/domain/src/logic/gain_loss.rs`.
6. Positions without a current price return `null` for current_value, gain_loss_absolute, and gain_loss_percentage.
7. Annotate endpoint with utoipa macros.

## Done criteria
- Positions are correctly calculated from buy/sell transactions.
- Closed positions (quantity = 0) are hidden by default, shown with `?show_closed=true`.
- Gain/loss is calculated correctly when current price is available.
- Positions without current price return null for value/gain fields.
- User can only access positions in their own portfolios.
- All clippy/fmt/doc/test checks pass.
