# Task 2.5: Transactions CRUD

## Objective
Implement full CRUD for transactions with multi-currency support, no-short-sell validation, and automatic asset creation.

## Steps
1. Create migration `migrations/0005_transactions.sql`:
   - `transactions` table per schema in MVP spec.
   - `portfolio_id` FK with ON DELETE CASCADE.
   - `asset_id` FK with ON DELETE RESTRICT.
   - Index on `(portfolio_id, asset_id)` and `(portfolio_id, date)`.
2. Create `crates/db/src/repositories/transaction_repo.rs`:
   - `create(new_transaction) -> Result<Transaction>`
   - `find_by_id(id) -> Result<Option<Transaction>>`
   - `find_by_portfolio(portfolio_id, pagination, filters) -> Result<PaginatedResult<Transaction>>` (filterable by transaction_type, asset_id, date range)
   - `update(id, fields) -> Result<Transaction>`
   - `delete(id) -> Result<()>`
   - `get_held_quantity(portfolio_id, asset_id) -> Result<Decimal>` — sum of buys minus sum of sells for a given asset in a portfolio.
3. Create `crates/api/src/transactions/`:
   - `dto.rs`:
     - `CreateTransactionRequest` — portfolio_id (from path), asset_isin, transaction_type, date, settlement_date, quantity, unit_price, commission, currency, gross_amount, tax_withheld, net_amount, notes. Garde validated.
     - Validation rules: Buy/Sell require quantity and unit_price. Dividend/Coupon require gross_amount and net_amount. Quantity must be positive.
     - `TransactionResponse` — all fields plus resolved asset name.
   - `handlers.rs`:
     - `GET /api/v1/portfolios/{id}/transactions` — list transactions for a portfolio. Verify portfolio ownership.
     - `POST /api/v1/portfolios/{id}/transactions` — create transaction. Verify portfolio ownership. Look up asset by ISIN — if not found, auto-create via asset_repo + OpenFIGI. For Sell: check held quantity, reject if insufficient (400 with InsufficientQuantity error).
     - `GET /api/v1/transactions/{id}` — get transaction. Verify ownership via portfolio.
     - `PUT /api/v1/transactions/{id}` — update transaction. Verify ownership. Re-validate no-short-sell if quantity or type changed.
     - `DELETE /api/v1/transactions/{id}` — delete transaction. Verify ownership. Check that deletion would not cause negative quantity for any subsequent sell transactions.
4. Exchange rate handling: if transaction currency != EUR, fetch current rate from Yahoo Finance (via price-fetcher) and store in `exchange_rate_to_base`. If fetch fails, require the rate in the request or reject.
5. All write operations (create, update, delete) must use `sqlx::Transaction` to ensure consistency.
6. Annotate all endpoints with utoipa macros.

## Done criteria
- Buy/Sell/Dividend/Coupon transactions can be created with correct fields.
- Sell exceeding held quantity returns 400 with clear error message.
- Creating a transaction with unknown ISIN auto-creates the asset.
- Deleting a transaction that would cause negative quantity for later sells is rejected.
- Transactions in non-EUR currency store the exchange rate.
- User can only access transactions in their own portfolios.
- Pagination and filters work correctly.
- All clippy/fmt/doc/test checks pass.
