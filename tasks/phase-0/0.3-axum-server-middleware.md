# Task 0.3: Axum Server, Health Endpoints, and Middleware

## Objective
Set up the Axum server with the full middleware stack and the /health and /ready operational endpoints.

## Steps
1. Add axum, tower-http, tokio dependencies to `crates/api/Cargo.toml`.
2. Create the Axum router in `crates/api/src/main.rs` with:
   - `GET /health` — returns 200 with `{ "status": "ok" }`.
   - `GET /ready` — checks database connectivity via a `SELECT 1` query. Returns 200 with `{ "status": "ready" }` if reachable, 503 with `{ "status": "unavailable" }` if not.
3. Configure the middleware stack (order matters, outermost first):
   - TraceLayer (request/response logging with request ID)
   - CompressionLayer (gzip/brotli)
   - CorsLayer (allowed origins from `CORS_ALLOWED_ORIGINS` env var)
   - RequestIdLayer + PropagateRequestIdLayer
   - TimeoutLayer (30 seconds)
   - Custom security headers middleware (X-Content-Type-Options: nosniff, X-Frame-Options: DENY, Referrer-Policy: no-referrer)
4. Configure tracing-subscriber with JSON structured output, reading log level from `RUST_LOG` env var.
5. Server binds to `SERVER_HOST:SERVER_PORT` from env (defaults: 0.0.0.0:3000).
6. Log startup message with server address and configured middleware.

## Done criteria
- `cargo run` starts the server (with PostgreSQL running).
- `curl http://localhost:3000/health` returns 200 + JSON.
- `curl http://localhost:3000/ready` returns 200 + JSON (database is up).
- Response headers include X-Content-Type-Options, X-Frame-Options, Referrer-Policy, and x-request-id.
- Structured JSON logs appear on stdout with request ID.
- Compressed response when `Accept-Encoding: gzip` is sent.
- All clippy/fmt/doc checks pass.
