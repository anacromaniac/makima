# Task 3.2: Pagination and Error Handling Review

## Objective
Ensure pagination and error handling are consistent across all endpoints.

## Steps
1. Pagination review:
   - Verify all list endpoints use the same `PaginatedResult<T>` wrapper with `{ "data", "pagination" }` format.
   - Verify default page=1, limit=25, max limit=100 is enforced everywhere.
   - Verify `total_items` and `total_pages` are correctly calculated in all repositories.
   - Check edge cases: page beyond total pages returns empty data (not error), limit=0 is rejected.
2. Error handling review:
   - Verify all error responses follow `{ "code": "...", "message": "..." }` format.
   - No endpoint leaks internal errors (SQL errors, stack traces, file paths) to the client.
   - Consistent error codes across all endpoints (e.g. VALIDATION_ERROR, NOT_FOUND, UNAUTHORIZED, INTERNAL_ERROR).
   - Garde validation errors are mapped to a readable format with field-level detail in the message.
3. Create a centralized error mapping in `crates/api/src/error.rs` if not already present:
   - Map `DomainError` variants to HTTP status codes and error codes.
   - Map sqlx errors to generic INTERNAL_ERROR (logged server-side, not exposed).
   - Map garde validation errors to VALIDATION_ERROR with field details.
4. Verify tracing: all errors are logged server-side with request ID before returning the generic response.

## Done criteria
- All list endpoints return identical pagination structure.
- All error responses follow the same format with no exceptions.
- No internal details leak in any error response.
- Every error is logged server-side with request ID.
- Validation errors include which field failed and why.
- All clippy/fmt/doc/test checks pass.

## Files modified
- `crates/api/src/error.rs` (created or updated)
- Various handler files for consistency fixes
